OpenVaultDB is an open protocol and platform for user-owned application databases. Apps connect to your vault and store data in namespaces they manage — but the vault, and everything in it, belongs to you.

The core principle
Applications do not own user data. Users own their data and grant applications permission to access it.

You decide where the vault lives — your laptop, a home server, a VPS, OpenVaultDB-hosted infrastructure, or your own cloud account. Each application gets an isolated namespace it fully controls: it creates its own schema, owns its migrations, and manages its own data lifecycle. No global data model. No lock-in.

What the protocol gives you

Ownership, portability, and a full audit trail — by design.

User-owned data

You own the OpenVaultDB instance. Applications receive credentials and scoped permissions — never the keys to your data.

Application namespaces

Each app gets an isolated area to create, alter, and delete its own schema and data. Apps own their migrations and versioning.

Bring your own storage

Self-host, use a managed tier, or plug in your own Firestore, Postgres, SQLite or cloud DB through DALgo storage adapters.

Git-backed history

The reference engine, inGitDB, versions every change — commit history, audit trail, and the ability to revert a record, table, or the whole vault.

Layered encryption

Encryption follows the deployment model. On third-party hosts, row-level encryption keeps contents unreadable to the provider by default.

Portable query API

DTQL exposes only portable functionality across every backend — so queries that work on Git work on Postgres, SQLite, or Firestore too.

No server required

Your browser writes straight to your own Git repo.

In serverless mode the app reads and writes your data directly from the browser to a Git repository you own — authenticated with the app's own GitHub sign-in, or a fine-grained token you provide. OpenVaultDB stays out of the data path entirely. inGitDB and INGR.io keep every change a clean, mergeable commit.

  • OpenVaultDB is never in the data path — your browser talks straight to GitHub
  • Private repos for private data, public repos for public data
  • Your data lives in a repo you control — clone it anywhere

Best for personal use and rarely-changing data. Need write throughput or scale? Run inGitDB on a server, or use a cloud database or RDBMS through DALgo — the right instrument for the job.

How it fits together

One protocol, many backends.

OpenVaultDB defines the protocol and APIs. Queries flow through DTQL into DALgo, the data abstraction layer, which maps them onto whichever storage adapter you've chosen. The reference path writes Git-friendly INGR.io files into a Git repository.

The ecosystem

A small family of focused, composable parts.

DTQL

Query language ↗

The primary API. A portable query and mutation language built on DALgo capabilities — intentionally not full SQL.

DALgo

Data abstraction layer ↗

Maps portable queries onto backend providers, so one query model spans many databases.

inGitDB

Git-backed engine ↗

The reference storage engine. Auditable history, easy rollback, and versioned data on top of Git.

INGR.io

Storage format ↗

A Git-friendly structured format designed for clean diffs, merges, and conflict resolution.

DataTug

Data explorer ↗

A generic browser and CLI for inspecting OpenVaultDB data — see exactly what each app stores.

Vault

Reference implementation ↗

Go libraries and the reference OpenVaultDB server — portable storage backends and DTQL query support. Run your own vault.

MVP scope

Honest about what's in, what's next, and what's later.

OpenVaultDB is an early proposal. Here's how the first milestone is scoped.

In the MVP

  • Authentication & token exchange
  • App namespaces
  • Schema management
  • DTQL & CRUD operations
  • DALgo integration
  • inGitDB backend
  • Version history
  • Basic web UI
  • DataTug integration
  • Self-hosting

Possibly included

  • SQLite backend
  • PostgreSQL backend
  • Firestore backend

Deferred

  • Advanced GraphQL
  • Complex joins
  • Stored procedures
  • Cross-app querying
  • Full federation
  • Advanced sharing model
Defined in the open

Specified, not improvised.

The OpenVaultDB protocol and its reference implementation are written as open, machine-readable specifications using the SpecScore.md standard. The spec is the source of truth — so any developer, any tool, and any AI agent can read it and build a conformant implementation.
