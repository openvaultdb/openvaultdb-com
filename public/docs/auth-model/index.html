Authentication Model

How a user connects an application to a vault they own, and how an application proves who it is. The guiding rule: the vault is the authority — OpenVaultDB only routes, and never becomes a party you're forced to trust.

Core decisions

Four rules hold the whole model together.

Each one exists to keep openvaultdb.com a convenience, never a dependency.

The account is optional

An OpenVaultDB account is a connection directory — a "vault wallet" that remembers your providers. It stores pointers and brokers consent. It never holds your data, and apps work without it.

The vault is the authority

The vault authenticates you and issues the app's scoped token. OpenVaultDB Connect only routes the app to the vault's authorize endpoint, then leaves the data path entirely.

The domain is the app's identity

An app's client_id is a domain it controls, proven by its redirect URI and a /.well-known/openvaultdb.yaml manifest. Namespaces are domain-bounded under it (<domain>/openvaultdb/<name>) — no central registry.

A manifest can delegate origins

The authoritative domain may list sibling origins in allowed_origins — so several mini-apps sharing one auth project (e.g. one Firebase Authentication) can act under one namespace.

Four authentication surfaces

OpenVaultDB's auth and an app's data auth never cross.

Authentication happens in four distinct places. The first three are OpenVaultDB's own, account-side, and are never exposed to an application. The fourth is the app's data token — and OpenVaultDB never sees or holds it.

#SurfaceProvidersExposed to app?
1Wallet loginAny — email, Google, GitHub, passkey…Never
2Owner loginAny (optional — not a registry)Never
3Vault registrationWallet login for the entry; GitHub optional, only to list reposNever
4App data accessVault server · app's own GitHub OAuth/App · user PATThis is the app's token
Adding a GitHub vault needs no GitHub sign-in. Surface 3 is just a [org/repo] text input — you type the repo. An optional "Sign in with GitHub to choose vault repo" picker can list your orgs and repos for convenience; it uses the token only to enumerate, then discards it and keeps a pointer. OpenVaultDB never validates the pointer, because it never touches the data.
The Connect flow

How an app gets access — and how OpenVaultDB steps out.

The happy path with a vault that is its own authority (a self-hosted or OpenVaultDB-hosted server).

1

The app opens OpenVaultDB Connect

A popup or redirect: "App sneat.app wants a namespace in your vault."

2

You choose a vault

Pick one from your directory — or paste a vault URL directly, which works with no account at all.

3

Connect routes you to that vault

OpenVaultDB redirects the app to the vault's own authorize endpoint. Its job is now essentially done.

4

The vault authenticates you and issues a token

The vault shows a consent screen (app name, logo, and scopes resolved from the manifest) and mints a token scoped to the app's namespace.

5

The app talks directly to the vault

From here the app uses that token straight against the vault. OpenVaultDB Connect is no longer involved.

GitHub-backed vaults — still no OpenVaultDB in the path. When the vault is a GitHub repo, GitHub is the authority. The app obtains its repo-scoped token through its own GitHub OAuth/App, or the user supplies a fine-grained PAT. OpenVaultDB never brokers it. (See the GitHub specifics below for why a backendless app uses a PAT.)
Where authority lives

One auth model, four deployment modes.

"The vault is the authority" resolves differently per backend — with exactly two explicit carve-outs.

Deployment modeAuthorityNotes
Self-hosted server The vault server Exposes an OAuth-style authorize/token endpoint and issues its own scoped tokens. Textbook.
OpenVaultDB-hosted The vault server Same as self-hosted; OpenVaultDB runs it as a service, but it is still a vault — not the broker.
Serverless (browser → GitHub) GitHub No vault server exists. The app gets a repo-scoped token via its own GitHub OAuth/App, or the user supplies a fine-grained PAT. OpenVaultDB is not involved.
User-owned cloud A vault server in front Raw cloud DBs (Firestore, DynamoDB…) have no namespace or consent concept, so this mode requires a thin vault server. Browser-direct-to-cloud is deliberately not a first-class mode — it would expose cloud credentials to the browser.
GitHub specifics

Three ways to land a repo-scoped token in the browser.

For a GitHub-backed vault, the app needs a token scoped to one repo. There are three interchangeable ways to get one — none of them route through OpenVaultDB.

Who brokersNeeds a backend?Best for
App's own GitHub AppYes — a minimal exchange endpointApps that already have GitHub OAuth. Prefer a GitHub App for per-repo scoping; use PKCE.
App's own OAuth AppYes — a minimal exchange endpointSame, but classic OAuth Apps grant the broad repo scope — less least-privilege.
User fine-grained PATNoFully backendless apps and personal use. The user pastes a token scoped to the repo; the browser uses it directly.
Why a backendless app must use a PAT. Verified against GitHub's docs (2025–2026): GitHub supports PKCE (since Jul 2025) and recommends it, but still requires a client secret for every client — it doesn't distinguish public from confidential clients, so PKCE alone can't replace a backend. And the token endpoint github.com/login/oauth/access_token sends no CORS headers, so a browser can't redeem a code directly. A self-brokering app therefore needs a tiny secret-holding endpoint; a truly backendless app uses a personal access token, which works because GitHub's REST data API is CORS-enabled for authenticated requests.
App identity

An app is its domain — and the manifest proves it.

No registration anywhere. The app publishes a manifest on the domain it controls; the namespaces it may claim are domain-bounded under that same domain (<domain>/openvaultdb/<name>).

# https://sneat.app/.well-known/openvaultdb.yaml
client_id: sneat.app
name: Sneat
logo: https://sneat.app/logo.svg
namespaces:            # each resolves to its own manifest.yaml
  - sneat.app/openvaultdb/calendar
  - sneat.app/openvaultdb/settings
allowed_origins:
  - https://sneat.app
  - https://datatug.app

Delegated origins

sneat.app owns the namespace, but vouches for datatug.app — a sibling app sharing the same Firebase Authentication. So datatug.app may connect on its behalf.

sneat.appauthoritative · owns namespace
─ vouches via allowed_origins ▶
datatug.appdelegated origin
Why it's safe. Only whoever controls sneat.app can edit its manifest. So a domain can pull other origins into its own namespace — but nothing can claim a namespace it isn't named by. A request from an unlisted origin is rejected.
Out of scope (for now)

Deliberately deferred

  • A verified app directory at openvaultdb.com — a convenience layer over domain identity, never required to connect.
  • Browser-direct-to-cloud as a first-class mode.
  • Table- and row-level permission scopes (namespace-level for now).
Still open

Questions we're working through

  • Serverless token handling in the browser — session vs. localStorage.
  • The exact scope string vocabulary.
  • Manifest caching, and how fast a delegated origin can be revoked.
  • Token format and lifetime — per-vault, or a minimum conformance profile.
